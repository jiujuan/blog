这是一个完整的电商核心交易链路，涵盖了从用户浏览商品到最终订单完成的全过程。

为了确保流程清晰，我将整个过程拆分为五个关键阶段，并分别提供详细步骤说明和流程图。

---

### 第一阶段：商品浏览与详情页 (Browsing & Detail)
**核心目标**：高性能读取，多级缓存保护，防止数据库压力。

#### 详细步骤：
1.  **用户请求**：用户在 APP/网页点击商品，发起 `GET /product/{id}` 请求。
2.  **网关层**：Nginx/网关进行限流、鉴权（如需），转发到商品服务。
3.  **L1 本地缓存**：应用服务器检查本地缓存（Caffeine/Guava）。
    *   *命中*：直接返回数据。
    *   *未命中*：进入下一步。
4.  **L2 分布式缓存**：查询 Redis。
    *   *命中*：回填 L1 缓存，返回数据。
    *   *未命中*：进入下一步。
5.  **数据库查询**：查询 MySQL 主库或读从库。
    *   *查到*：写入 Redis（设置随机 TTL），写入 L1，返回数据。
    *   *未查到*：写入 Redis 空值（防穿透，短 TTL），返回“商品不存在”。
6.  **渲染展示**：前端接收数据，渲染商品详情（价格、库存概览、图片）。

#### 流程图：
```text
[用户] 
   |
   v
[API 网关] (限流/鉴权)
   |
   v
[商品服务]
   |---> [L1 本地缓存] --(命中)--> [返回数据]
   |        |
   |      (未命中)
   |        v
   |---> [L2 Redis 缓存] --(命中)--> [回填 L1] --> [返回数据]
   |        |
   |      (未命中/击穿)
   |        v
   |---> [互斥锁?] --(获取失败)--> [休眠重试]
   |        |
   |      (获取成功)
   |        v
   |---> [MySQL 数据库] --(有数据)--> [写 Redis + 写 L1] --> [返回数据]
   |        |
   |      (无数据)
   |        v
   |---> [写 Redis 空值] --> [返回 404]
   |
   v
[前端页面渲染]
```

### 第二阶段：下单前置校验 (Pre-Order Validation)
**核心目标**：确认购买资格，锁定库存（预占），生成订单号。
*注意：此时尚未真正扣减库存，而是“冻结”库存，防止超卖。*

#### 详细步骤：
1.  **提交订单请求**：用户点击“立即购买”，携带 `SKU ID`, `数量`, `收货地址ID` 等。
2.  **基础校验**：
    *   用户状态是否正常？
    *   商品是否上架？
    *   购买数量是否超过限购？
    *   收货地址是否有效？
3.  **库存预占 (Try 阶段)**：
    *   调用库存服务，执行 `UPDATE stock SET frozen = frozen + num WHERE id = sku AND stock - frozen >= num`。
    *   *失败*：返回“库存不足”，流程终止。
    *   *成功*：获得库存锁定凭证。
4.  **价格计算**：根据优惠券、会员折扣、运费模板计算最终支付金额（防止前端篡改价格）。
5.  **生成订单号**：使用雪花算法（Snowflake）生成全局唯一订单 ID。
6.  **创建订单记录**：在订单库插入一条状态为 `UNPAID` (待支付) 的记录。
7.  **返回收银台**：向前端返回订单号和应付金额，引导去支付。

#### 流程图：
```text
[用户点击"立即购买"]
   |
   v
[订单服务 - 前置校验]
   |---> [校验用户状态/登录]
   |---> [校验商品状态/上下架]
   |---> [校验限购规则]
   |
   v
[库存服务 - 预占库存 (Try)]
   |---> SQL: UPDATE stock SET frozen=frozen+N WHERE available>=N
   |        |
   |      (失败: 库存不足) ---> [返回错误: 商品售罄] ---> [结束]
   |        |
   |      (成功: 锁定成功)
   |
   v
[订单服务 - 计价与落库]
   |---> [计算最终金额 (扣除优惠/运费)]
   |---> [生成唯一订单号 (Snowflake)]
   |---> [DB: 插入订单表 (Status=UNPAID)]
   |
   v
[返回收银台页面] (显示: 订单号, 金额, 倒计时 30 分钟)
```

### 第三阶段：订单支付 (Payment Processing)
**核心目标**：资金安全，状态流转，幂等性处理。

#### 详细步骤：
1.  **发起支付**：用户在收银台选择支付方式（微信/支付宝/银行卡），点击支付。
2.  **请求支付网关**：订单服务调用内部支付服务，或直接对接三方支付渠道。
3.  **跳转/唤起支付**：前端唤起三方支付 APP 或扫描二维码。
4.  **用户付款**：用户在第三方完成密码验证/指纹验证。
5.  **异步回调 (Webhook)**：
    *   第三方支付平台向商户服务器发送 `POST /notify` 回调通知。
    *   **验签**：验证签名，确保请求来自官方。
    *   **幂等检查**：检查该订单是否已处理过（防止重复入账）。
    *   **金额核对**：核对回调金额与订单金额是否一致。
6.  **更新订单状态**：
    *   将订单状态从 `UNPAID` 更新为 `PAID` (已支付)。
    *   记录支付流水号、支付时间。
7.  **响应第三方**：返回 `SUCCESS` 给支付平台，否则平台会持续重试回调。
8.  **前端轮询/推送**：前端通过 WebSocket 或轮询检测到支付成功，跳转“支付成功页”。

#### 流程图：
```text
[用户] --> [选择支付方式] --> [唤起三方支付 (微信/支付宝)]
                                   |
                                   v
                            [用户输入密码/指纹]
                                   |
                                   v
                            [三方支付处理资金]
                                   |
                                   | (异步通知)
                                   v
[商户支付服务] <------- [支付平台回调 (Webhook)]
   |
   |---> [1. 验签 (Signature Check)]
   |---> [2. 幂等检查 (是否已处理?)] --(是)--> [直接返回 SUCCESS]
   |---> [3. 金额核对]
   |
   v
[订单服务 - 状态流转]
   |---> [DB: UPDATE order SET status=PAID WHERE id=? AND status=UNPAID]
   |---> [记录支付流水]
   |
   v
[返回 SUCCESS 给支付平台] (停止回调)
   |
   v
[前端: 轮询/WSPush 收到成功信号] --> [跳转"支付成功"页]
```

### 第四阶段：库存实扣与履约 (Inventory Deduction & Fulfillment)
**核心目标**：将“预占”转为“实扣”，触发发货流程。
*此步骤通常在支付成功后立即触发（同步或异步消息）。*

#### 详细步骤：
1.  **触发扣减**：
    *   方式 A（同步）：在支付回调成功后的代码逻辑中，直接调用库存服务。
    *   方式 B（异步 - 推荐）：支付成功后发送 MQ 消息 `OrderPaidEvent`，库存服务消费消息。
2.  **库存实扣 (Confirm 阶段)**：
    *   库存服务接收请求/消息。
    *   执行逻辑：`available_stock = available_stock - num`，`frozen_stock = frozen_stock - num`。
    *   *注意*：此时才是真正的库存减少。
3.  **异常处理**：
    *   如果实扣失败（极罕见，如数据损坏），需触发报警或人工介入，甚至触发回滚（退款）。
4.  **触发履约**：
    *   订单状态更新为 `PAID_AND_STOCK_DEDUCTED`。
    *   推送消息给**仓储系统 (WMS)**，生成发货单，开始拣货打包。

#### 流程图：
```text
[支付成功事件]
   |
   +---> (方式 A: 同步调用) ------------------+
   |                                         |
   +---> (方式 B: 发送 MQ 消息)              |
            |                                |
            v                                |
      [MQ: OrderPaidTopic]                   |
            |                                |
            v                                |
   [库存服务] <------------------------------+
   |
   |---> [校验预占记录是否存在]
   |---> [DB: 实扣库存]
   |     UPDATE stock 
   |     SET available = available - num, 
   |         frozen = frozen - num 
   |     WHERE id = sku AND frozen >= num
   |
   |---> (成功) --> [更新订单状态: 已扣库] --> [推送 WMS 生成发货单]
   |
   |---> (失败) --> [记录异常日志] --> [触发报警/人工介入] --> [可能触发自动退款]
```

### 第五阶段：全链路异常补偿 (Exception Handling)
**核心目标**：处理超时未支付、取消订单等情况，释放资源。

#### 详细步骤：
1.  **超时未支付**：
    *   利用**延迟队列**（如 RocketMQ 延迟消息、Redis Key 过期监听、时间轮）在 30 分钟后触发检查。
    *   若订单仍为 `UNPAID`，执行**关单逻辑**。
2.  **用户主动取消**：
    *   用户点击“取消订单”。
3.  **释放库存 (Cancel 阶段)**：
    *   调用库存服务：`frozen_stock = frozen_stock - num`，`available_stock = available_stock + num`。
    *   更新订单状态为 `CANCELLED`。
4.  **退款处理**：
    *   若已支付后发生退货/退款，走逆向流程：WMS 收货 -> 触发退款 -> 库存回滚（入库）。

#### 流程图 (关单/取消流程)：
```text
[触发源: 超时延迟队列 OR 用户主动取消]
   |
   v
[订单服务 - 关单逻辑]
   |---> [DB: 检查订单状态 (必须是 UNPAID 或 PAID_申请退款)]
   |---> [DB: 更新订单状态为 CANCELLED]
   |
   v
[库存服务 - 释放库存]
   |---> [DB: 释放预占/回滚库存]
   |     UPDATE stock 
   |     SET frozen = frozen - num, 
   |         available = available + num
   |
   v
[结束流程] (发送通知: "订单已关闭")2332
```

### 关键技术点回顾
1.  **缓存一致性**：浏览阶段使用 `Local + Redis` 多级缓存。
2.  **防超卖**：下单阶段使用 `SQL 乐观锁` 或 `Redis Lua 脚本` 进行库存预占。
3.  **分布式事务**：支付与扣库之间使用 `MQ 事务消息` 或 `TCC` 保证最终一致性。
4.  **幂等性**：支付回调、库存扣减接口必须设计幂等，防止网络重试导致数据错误。
5.  **超时处理**：利用延迟队列实现自动关单和库存释放。
